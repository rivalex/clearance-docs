# Введение

## Что такое Clearance?

Clearance - это готовая к использованию панель администратора для приложений Laravel, построенная поверх [spatie/laravel-permission](https://github.com/spatie/laravel-permission). Она добавляет панель на Livewire 4 + Flux UI для управления разрешениями, ролями, guard'ами и настройками, а также слой контекстной авторизации и потолки (ceiling) для ролей - устанавливается одной командой artisan, без изменения каких-либо таблиц Spatie.

Clearance даёт вам:

- **Панель администратора** для Разрешений, Ролей, Guards, Настроек и Дашборда - используйте готовые маршруты или встраивайте каждый Livewire-компонент по отдельности.
- **Контекстная авторизация** - выдавайте роль или разрешение в привязке к конкретному экземпляру модели (Store, Tenant, Project) с помощью `$user->canIn()`, директив Blade `@canin`/`@hasrolein` и `ContextService` для использования вне модели User.
- **Ceiling-роли** - роль может объявить родительскую роль, чьё множество разрешений выступает верхней границей; дочерняя роль никогда не может превысить родительскую - это обеспечивается автоматически при каждом сохранении.
- **Точечные права на запись** - одно делегируемое разрешение `clearance-{section}-write` на каждый раздел панели, так что вы можете выдать "управление ролями", не выдавая заодно "управление настройками".
- **Super Admin** с опциональным глобальным обходом Gate, обнаружением алиасов для уже существующих admin-подобных ролей и безопасным неразрушающим повышением роли.
- **Модуль пользователей** (опционально) - назначайте роли глобально или в конкретном контексте, плюс переопределения разрешений по контексту, из панели для отдельного пользователя.
- **Панель настроек** - отображаемое имя, описание, санитизированная SVG-иконка и цвет для каждой роли/guard'а; настраиваемая роль по умолчанию с авто-назначением и массовым назначением.
- **Управление Guards** - guard'ы, хранящиеся в базе данных, внедряются в `auth.guards` при загрузке приложения, ограничены allow-list'ом драйверов.
- **9 языков в комплекте** (ar, en, es, fr, hi, it, pt, ru, zh), паритет ключей проверяется тестами.
- **Точки расширения** - трейт `HasClearance`, трейт `HasPermissionGroups`, контракт `ClearanceContextable`, переиспользуемые компоненты Blade.
- **Санитизация SVG** - каждая иконка, загруженная пользователем, проходит через санитайзер на основе allow-list перед сохранением и при рендеринге.
- **Защищено с точки зрения безопасности** - ограничения (ceiling) против повышения привилегий на каждом пути выдачи разрешений, приоритет запрета в контекстных переопределениях, защищённое пространство имён `clearance-*`. См. [Безопасность](/ru/guide/security).

## Связь с spatie/laravel-permission

Clearance - это не замена пакета разрешений Spatie, а слой UI и авторизации поверх него. Роли и разрешения остаются моделями `Role` и `Permission` из Spatie и по-прежнему хранятся в таблицах Spatie `roles`, `permissions`, `model_has_roles`, `model_has_permissions` и `role_has_permissions`. Clearance никогда не изменяет эти таблицы. Всё, что Clearance добавляет - метаданные отображения, область роли, ceiling, контекстные назначения, переопределения по контексту - хранится в собственном наборе таблиц `clr_*`. См. [Архитектуру](/ru/guide/architecture) для полного разбора.

## Ключевые понятия

| Понятие | Описание |
|---|---|
| Permission (разрешение) | Разрешение Spatie, следующее соглашению об именовании `group-action` (напр. `orders-create`). Предоставляет способность (ability). |
| Область роли (role scope) | Каждая роль либо `global` (назначается напрямую, применяется везде), либо `contextual` (привязана к конкретным типам моделей, назначается только в рамках экземпляра контекста). |
| Ceiling-роль | Роль может объявить родительскую роль, чьё множество разрешений является верхней границей - дочерняя роль никогда не может её превысить. |
| Контекст | Экземпляр модели (Store, Tenant, Project), к которому может быть привязана роль или переопределение разрешения. |
| Переопределение (override) | Персональное, привязанное к контексту разрешение или запрет `forced_on`/`forced_off` - запрет всегда побеждает. |

Для каждого из этих понятий есть отдельная страница: [Разрешения](/ru/features/permissions), [Роли](/ru/features/roles), [Контекстная авторизация](/ru/features/contextual-authorization), [Пользователи](/ru/features/users), [Super Admin](/ru/features/super-admin).
