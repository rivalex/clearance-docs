# Безопасность

Clearance прошёл выделенный аудит безопасности перед публикацией, а также два дополнительных этапа усиления защиты перед релизом. Основные моменты: ограничения (ceiling) против повышения привилегий на каждом пути выдачи разрешений (как при прямой выдаче, так и при синхронизации на уровне роли), приоритет запрета в контекстных переопределениях, санитайзер SVG на основе allow-list, защищённое пространство имён разрешений `clearance-*` и безопасные для гостя директивы Blade.

## Доступ к панели и точечные права на запись

Доступ на чтение ко всей панели требует разрешения `clearance-access` (проверяется через `can()`, никогда через `hasRole()`). Операции записи в каждом разделе защищены отдельным делегируемым разрешением - все они создаются `clearance:install` и назначаются `super_admin`:

| Разрешение | Контролирует |
|---|---|
| `clearance-permissions-write` | Создание / редактирование / удаление разрешений |
| `clearance-roles-write` | Создание / редактирование / удаление ролей и их назначений разрешений |
| `clearance-guards-write` | Создание / редактирование / удаление guard'ов |
| `clearance-settings-write` | Редактирование метаданных отображения, иконок, цветов, роли по умолчанию |
| `clearance-users-write` | Назначение / удаление ролей для конкретного пользователя |

Если строка точечного разрешения вообще не существует в базе данных (устаревшее состояние до появления seed'а), проверка откатывается к `clearance-access` для обратной совместимости.

## Защищённое пространство имён `clearance-*`

Разрешения `clearance-*` зарезервированы: их нельзя создавать, переименовывать, удалять или добавлять к какой-либо роли иначе как через собственный процесс установки пакета - даже актором `super_admin` для самой роли `super_admin`.

## Ограничения против повышения привилегий

Каждый путь выдачи разрешений в Clearance применяет ceiling, так что актор никогда не может выдать больше, чем имеет сам:

- **Ceiling-роли** - роль может объявить родительскую роль, чьё множество разрешений выступает верхней границей. Любое избыточное разрешение автоматически и незаметно обрезается при `RoleService::setParent()` и при каждой последующей синхронизации разрешений - без исключения, без сообщения об ошибке. Удаление разрешений у родителя автоматически каскадно распространяется на каждую дочернюю роль. См. [Роли](/ru/features/roles) для полного API ceiling.
- **Модуль пользователей** - и назначение ролей, и прямая/переопределяющая выдача разрешений применяют ограничение по привилегиям: актор, не являющийся `super_admin`, не может изменять собственные назначения и не может выдать разрешение, которым не обладает сам. См. [Пользователи](/ru/features/users).

## Приоритет запрета в контекстных переопределениях

Переопределения разрешений по контексту (`clr_ctx_overrides`) поддерживают `forced_on` и `forced_off`. `forced_off` всегда побеждает - даже над разрешением, выданным глобальной ролью. Не существует пути, позволяющего выдаче с более низким приоритетом перекрыть явный запрет. См. [Архитектуру](/ru/guide/architecture#how-resolution-works) для полного порядка разрешения (resolution).

## Санитизация SVG

Каждая предоставленная пользователем иконка (метаданные отображения роли/guard'а в панели Настроек) проходит через санитайзер на основе allow-list как перед сохранением, так и при рендеринге - а не единоразовую проверку при загрузке.

## Безопасные для гостя директивы Blade

`@canin` и `@hasrolein` обе безопасно разрешаются (возвращают `false`) для гостевого пользователя - они никогда не выбрасывают исключение.

## Обход Gate действует во всём приложении

`super_admin_gate_bypass` (по умолчанию `false`) управляет тем, обходит ли роль `super_admin` все проверки `can()` / `Gate::allows()` / политик Laravel глобально, через хук `Gate::before()`.

::: warning Не ограничено рамками панели
Если включено, любой пользователь, обладающий `super_admin`, пропускает все проверки Gate во всём приложении - не только проверки панели Clearance. Включайте это осознанно. См. [Super Admin](/ru/features/super-admin) для поведения обнаружения алиасов и повышения роли.
:::

## Нюансы, важные для безопасности

- `forced_off` всегда побеждает - даже над разрешением, выданным глобальной ролью.
- Ceiling-роль только обрезает, но никогда не выдаёт - установка разрешительного родителя сама по себе не даёт дочерней роли ничего.
- Обход Gate (`super_admin_gate_bypass`) действует во всём приложении, а не только в рамках панели.
- Модуль пользователей по умолчанию выключен (`modules.users = false`).
- Разрешения `clearance-*` неприкосновенны вне процесса установки, даже для `super_admin`.
- Всегда запускайте `clearance:install`, а не обычный `migrate`, на новой установке - см. [Установку](/ru/guide/installation).
