# Super Admin

Роль `super_admin` создаётся автоматически командой `clearance:install` и поддерживается в синхронизации со всеми разрешениями `clearance-*`.

```bash
# Install and make user ID 1 a super admin
php artisan clearance:install --user=1
```

См. [руководство по установке](/ru/guide/installation) и [справочник команд Artisan](/ru/api/artisan-commands) для полного списка флагов установщика.

## Обход Gate (опционально)

`super_admin_gate_bypass` управляет тем, обходит ли роль `super_admin` все проверки `can()` / `Gate::allows()` / политик Laravel глобально, через хук `Gate::before()`. По умолчанию: `false`. Обход опционален. Чтобы включить его:

```php
// config/clearance.php
'super_admin_gate_bypass' => true,
```

::: danger Действует во всём приложении, а не только в панели
Если включено, любой пользователь, обладающий `super_admin`, пропускает **все** проверки Gate во всём приложении - не только проверки панели Clearance. Это затрагивает каждую проверку `can()`, `Gate::allows()` и политики в вашем приложении, а не только те, что определяет Clearance. Для более узкой области оставьте это выключенным и выдавайте разрешения явно.
:::

## Обнаружение алиасов

Если в вашей базе данных уже есть роль с именем `super_admin`, `superadmin`, `super-admin`, `root` или `owner`, установщик обнаруживает её и интерактивно предлагает либо повысить её, либо создать отдельную роль `super_admin`. Используйте `--super-admin-role=NAME`, чтобы пропустить запрос:

```bash
php artisan clearance:install --super-admin-role=owner
```

::: tip Повышение неразрушающее
Повышение переименовывает роль на месте - все существующие назначения `model_has_roles` сохраняются благодаря внешнему ключу на `role_id`. Ни один пользователь не теряет существующее назначение роли во время повышения.
:::

## Доступ к панели и точечные права на запись

Перейдите на `/clearance` после установки. Доступ на чтение ко всей панели требует разрешения `clearance-access`, проверяемого через `can()`, никогда через `hasRole()`:

```php
$user->givePermissionTo('clearance-access');
```

Операции записи в каждом разделе защищены отдельным делегируемым разрешением - все они создаются `clearance:install` и назначаются `super_admin`:

| Разрешение | Контролирует |
|---|---|
| `clearance-permissions-write` | Создание / редактирование / удаление разрешений |
| `clearance-roles-write` | Создание / редактирование / удаление ролей и их назначений разрешений |
| `clearance-guards-write` | Создание / редактирование / удаление guard'ов |
| `clearance-settings-write` | Редактирование метаданных отображения, иконок, цветов, роли по умолчанию |
| `clearance-users-write` | Назначение / удаление ролей для конкретного пользователя |

```php
// Grant a user write access to roles only:
$user->givePermissionTo('clearance-access');
$user->givePermissionTo('clearance-roles-write');
```

::: tip Предоставление узкого доступа
Скомбинируйте `clearance-access` ровно с одним разрешением `clearance-{section}-write`, чтобы дать пользователю (скажем, тимлиду) доступ на запись только к одному разделу панели - например, к ролям - не открывая при этом настройки, guard'ы или управление пользователями.
:::

Если строка точечного разрешения вообще не существует в базе данных (устаревшее состояние до появления seed'а), проверка откатывается к `clearance-access` для обратной совместимости. На любом экземпляре, где уже был выполнен `clearance:install`, этот откат никогда не активируется.

## Зарезервированные разрешения `clearance-*`

Разрешения `clearance-*` зарезервированы: их нельзя создавать, переименовывать, удалять или добавлять к какой-либо роли иначе как через собственный процесс установки пакета - даже актором `super_admin` для самой роли `super_admin`.
