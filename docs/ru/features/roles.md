# Роли

Панель Ролей - это полноценный CRUD-интерфейс над моделью `Role` из Spatie, расширенный двумя специфичными для Clearance понятиями: **область (scope)** (глобальная или контекстная) и **ceiling** (родительская роль, ограничивающая разрешения дочерней).

## Что делает панель

- **Создание / редактирование / удаление** ролей.
- **Назначение разрешений в рамках guard'а** - назначение разрешений роли, отфильтрованных по guard'у роли.
- **Выбор области (scope)** - `global` или `contextual`.
- **Настройка ceiling** - установка или удаление родительской роли.

## Область: глобальная или контекстная

Каждая роль объявляет область при создании:

- **`global`** - назначается пользователю напрямую через `$user->assignRole()`, применяется везде.
- **`contextual`** - привязана к одному или нескольким типам моделей (`context_types`), назначается только в рамках конкретного экземпляра контекста через `UserRoleContext`. См. [Контекстную авторизацию](/ru/features/contextual-authorization) и [Пользователи](/ru/features/users#contextual-roles-under-the-hood).

Роли без явной строки `RoleMeta` по умолчанию считаются `global` для обратной совместимости.

## Ceiling-роли

Роль может объявить ceiling - родительскую роль, чьё множество разрешений выступает верхней границей:

```php
use Rivalex\Clearance\Services\RoleService;

app(RoleService::class)->setParent($childRole, $parentRole);
```

<Screenshot src="/screenshots/role-form-ceiling.png" alt="Форма роли с выбранной родительской ceiling-ролью" caption="Установка родительской ceiling-роли в форме роли (RoleForm)." />

Правила, применяемые автоматически при каждом сохранении:

- Дочерняя роль может обладать только теми разрешениями, которыми обладает и родительская. Любой избыток незаметно обрезается при `setParent()` и при каждой последующей синхронизации разрешений - без исключения, без сообщения об ошибке, просто применяется автоматически.
- Роль, которая уже выступает родителем, не может стать дочерней (`ClearanceScopeViolationException`).
- Роль, которая уже является дочерней, не может сама выступать родителем (`ClearanceScopeViolationException`).
- Удаление разрешений у родителя автоматически каскадно распространяется на каждую дочернюю роль.

::: warning Ceiling только обрезает, но никогда не выдаёт
Установка разрешительного родителя сама по себе не даёт дочерней роли ничего. Ceiling - это строго верхняя граница: он никогда не добавляет разрешения дочерней роли, а может только убирать их.
:::

Это настраивается для каждой роли в панели Ролей (`RoleForm`) или программно через `RoleService::setParent()` / `RoleService::removeParent()`.

## Контроль доступа

Доступ на чтение требует `clearance-access`. Создание, редактирование или удаление роли - включая её назначения разрешений, область и ceiling - требует `clearance-roles-write`:

```php
// Grant a user write access to roles only:
$user->givePermissionTo('clearance-access');
$user->givePermissionTo('clearance-roles-write');
```

## Маршрут и встроенный компонент

| URI | Имя маршрута | Описание |
|---|---|---|
| `/clearance/roles` | `clearance.roles` | CRUD ролей с назначением разрешений в рамках guard'а, областью (глобальная/контекстная) и ceiling |

```blade
<livewire:clearance::roles.role-manager />
```

## База данных

Метаданные роли (область, ceiling, состояние блокировки) хранятся в `clr_role_meta`, никогда в таблице `roles` Spatie:

| Таблица | Ключевые столбцы |
|---|---|
| `clr_role_meta` | `role_id` (уникален, FK -> `roles`) - `is_locked`, `scope` (`global`\|`contextual`), `context_types` (json) - `parent_role_id` (nullable FK -> `roles`, `ON DELETE SET NULL`) |
