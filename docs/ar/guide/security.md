# الأمان

خضعت Clearance لتدقيق أمني مخصص قبل النشر بالإضافة إلى جولتين متابعة لتحصين إضافي قبل الإصدار. أبرز النقاط: حدود سقفية لمنع تصعيد الصلاحيات على كل مسار يمنح صلاحية (المنح المباشر ومزامنة الصلاحيات على مستوى الدور على حد سواء)، وتجاوزات سياقية ذات أولوية للرفض، ومُعقِّم SVG بقائمة سماح، ومساحة أسماء `clearance-*` محمية، وتوجيهات Blade آمنة للضيوف.

## الوصول إلى اللوحة وصلاحيات الكتابة الدقيقة التحكم

يتطلب الوصول للقراءة إلى اللوحة بأكملها صلاحية `clearance-access` (يُتحقق منها عبر `can()`، وليس أبداً عبر `hasRole()`). عمليات الكتابة على كل قسم مقيّدة بصلاحية مخصصة قابلة للتفويض - جميعها تُنشأ بواسطة `clearance:install` وتُعيَّن لـ `super_admin`:

| الصلاحية | تتحكم في |
|---|---|
| `clearance-permissions-write` | إنشاء / تعديل / حذف الصلاحيات |
| `clearance-roles-write` | إنشاء / تعديل / حذف الأدوار وتعييناتها الخاصة بالصلاحيات |
| `clearance-guards-write` | إنشاء / تعديل / حذف الحراس |
| `clearance-settings-write` | تعديل بيانات العرض الوصفية، والأيقونات، والألوان، والدور الافتراضي |
| `clearance-users-write` | تعيين / إزالة الأدوار لمستخدم محدد |

إذا لم يكن صف صلاحية دقيق التحكم موجوداً على الإطلاق في قاعدة البيانات (حالة قديمة سابقة للتزويد)، يعود الفحص إلى `clearance-access` للتوافق مع الإصدارات السابقة.

## مساحة أسماء `clearance-*` المحمية

صلاحيات `clearance-*` محجوزة: لا يمكن إنشاؤها، أو إعادة تسميتها، أو حذفها، أو إضافتها إلى أي دور إلا عبر تدفق التثبيت الخاص بالحزمة نفسها - حتى بواسطة ممثل (actor) من نوع `super_admin` لدور `super_admin` نفسه.

## حدود سقفية لمنع تصعيد الصلاحيات

كل مسار يمنح صلاحية في Clearance يفرض سقفاً بحيث لا يمكن لممثل (actor) أن يمنح أكثر مما يحمله هو نفسه:

- **الأدوار السقفية (Ceiling roles)** - يمكن لدور ما أن يعلن عن دور أب تُشكّل مجموعة صلاحياته حداً أعلى. يُقلَّم أي فائض من الصلاحيات بصمت عند `RoleService::setParent()` وعند كل مزامنة صلاحيات لاحقة - دون استثناء، ودون رسالة خطأ. إزالة صلاحيات من دور أب تتسلسل الإزالة إلى كل دور فرعي تلقائياً. انظر [الأدوار](/ar/features/roles) لواجهة الـ API الكاملة للسقف.
- **وحدة المستخدمين** - يفرض تعيين الأدوار ومنح الصلاحيات المباشر/التجاوزات على حد سواء سقفاً لمنع تصعيد الصلاحيات: لا يمكن لممثل غير `super_admin` تعديل تعييناته الخاصة، ولا يمكنه منح صلاحية لا يحملها هو نفسه. انظر [المستخدمون](/ar/features/users).

## تجاوزات سياقية ذات أولوية للرفض

تدعم التجاوزات السياقية للصلاحيات (`clr_ctx_overrides`) نوعي `forced_on` و`forced_off`. تفوز `forced_off` دائماً - حتى على صلاحية ممنوحة من دور عام. لا يوجد مسار يسمح لمنح ذي أولوية أقل بتجاوز رفض صريح. انظر [البنية المعمارية](/ar/guide/architecture#how-resolution-works) لترتيب التحليل الكامل.

## تعقيم SVG

كل أيقونة يقدّمها المستخدم (بيانات عرض الدور/الحارس الوصفية في لوحة الإعدادات) تمر عبر مُعقِّم بقائمة سماح قبل التخزين وعند العرض على حد سواء - وليس مجرد فحص لمرة واحدة عند الرفع.

## توجيهات Blade آمنة للضيوف

تُحلَّل كل من `@canin` و`@hasrolein` بأمان (تُرجع `false`) لمستخدم ضيف - ولا تُطلقان استثناءً أبداً.

## تجاوز Gate على مستوى التطبيق بأكمله

يتحكم `super_admin_gate_bypass` (الافتراضي `false`) فيما إذا كان دور `super_admin` يتجاوز جميع فحوصات `can()` / `Gate::allows()` / السياسات (policies) الخاصة بـ Laravel على مستوى التطبيق بأكمله، عبر خطاف (hook) من نوع `Gate::before()`.

::: warning ليس مقيّداً باللوحة
عند التفعيل، يتخطى أي مستخدم يحمل `super_admin` جميع فحوصات Gate على مستوى التطبيق بأكمله - وليس فقط فحوصات لوحة Clearance. فعّله عن قصد. انظر [المشرف الفائق](/ar/features/super-admin) لسلوك كشف الأسماء البديلة والترقية.
:::

## نقاط دقيقة ذات صلة بالأمان

- تفوز `forced_off` دائماً - حتى على صلاحية ممنوحة من دور عام.
- الدور السقفي يُقلّم فقط، ولا يمنح أبداً - تعيين دور أب متساهل لا يمنح الدور الفرعي أي شيء بحد ذاته.
- تجاوز Gate (`super_admin_gate_bypass`) هو على مستوى التطبيق بأكمله، وليس مقيّداً باللوحة.
- وحدة المستخدمين معطّلة افتراضياً (`modules.users = false`).
- صلاحيات `clearance-*` غير قابلة للمس خارج تدفق التثبيت، حتى بالنسبة لـ `super_admin`.
- شغّل دائماً `clearance:install`، وليس `migrate` مباشرة، على إعداد جديد - انظر [التثبيت](/ar/guide/installation).
